# Improving the asp-arc pipeline: diagnosis + ranked experiments

## Context

The asp-arc pipeline asks Nemotron-Cascade-2 to produce Clingo/ASP programs that solve ARC-AGI grid-transformation puzzles. The current bottleneck is **syntactic, not semantic**: the model rarely produces a Clingo-valid program, and the refinement loop almost never escapes the syntax-error regime to even see the actual semantic problem.

Evidence (88 puzzle audits across 11 runs in `src/audit/2026042*/`):
- **0/88 puzzles solved** end-to-end on these baseline runs.
- Of 1,435 train-example verifications across all attempts: **1,264 clingo_error**, 97 unsatisfiable, 74 wrong_values. Syntax/grounding failures are ~88% of all observed failures.
- Refinement is essentially stuck: **295/361 (82%) of attempt-to-attempt transitions remain `clingo_error → clingo_error`**. Only a small fraction of refinements move into the semantic regime.
- The agentic-nemotron branch (multi-candidate + syntax-agent) has demonstrated isolated wins (e.g. puzzle `8d510a79` solved at refinement 5) but is heavyweight; the user wants experiments built on top of `feature/baseline`.

The goal of this plan is to (a) name the actual failure modes with evidence, and (b) propose a ranked list of independent, single-PR experiments on `feature/baseline` that each test one hypothesis with a clear metric.

## Diagnosis: what the model gets wrong

### Top error categories (audit-counted)

| Rank | Category | Verifs hit | Representative error |
|---|---|---|---|
| 1 | **Unsafe variables** in aggregates / heads | 499 (1,439 lines) | `error: unsafe variables in: total_rows(N,R) :- N = #count{R1 : input(R1,_,_)}.` — `R` appears in head but is not bound by the body. |
| 2 | **`unexpected =`** — aggregate on left, or assignment used as a head | 242 (626 lines) | `#min{R : nonzero(R,_,_)} = MR` (aggregate-on-left) or `Var = expr` as a fact. |
| 3 | **`unexpected #count` / `#max` / `#sum`** — aggregate placed where atom expected | 187+ | `output(R,C, #count{...})` — aggregates can't be argument terms. |
| 4 | **`unexpected ,` expecting `}` or `;`** — malformed aggregate body | 135 | `#count{R, C : input(R, C, _)}` (extra term in tuple) instead of `#count{(R,C) : input(R,C,_)}`. |
| 5 | **`unexpected :-`** — stray / orphan colons in rule continuations | 159 | Indented continuation lines starting with `:-`. |
| 6 | **`unexpected #count`** + comparison ops (`<`, `>`, `!=`) | 250+ | Comparisons embedded inside aggregate term position. |
| 7 | **Lexer error: backtick** | 57 | Markdown ``` ` ``` characters surviving extraction. |
| 8 | **Lexer error: `#mod`** | 36 | Model still emits `#mod` despite docs saying `\`. |
| 9 | **Empty programs** | 39 | Code extraction returned nothing usable. |
| 10 | **Unsatisfiable / wrong_values** (semantic) | 171 total | The model *did* produce valid ASP but the logic is wrong. |

### Root causes (synthesized)

1. **Aggregate semantics are the dominant obstacle.** Categories 1–3 and 6 are all aggregate-related and together account for the majority of errors. The model treats `#count{...}` like a numeric expression, places it on the LHS of `=`, embeds it inside atom arguments, and fails to bind aggregate-introduced variables in the head. The current `quick_syntax_fix` regexes (on `feature/agentic-nemotron`) catch a handful of these cases but the model regenerates the same patterns on the next attempt.
2. **Refinement feedback does not reframe the mistake.** With current prompts, when the model re-attempts after a syntax error it tends to repeat the same shape (audit transitions: 82% stay in `clingo_error`). Feedback shows the Clingo error verbatim but does not point at the *concept* that's wrong (e.g. "you used an aggregate as a term — extract it into a defining rule first").
3. **Code extraction leaks markup.** Backtick lexer errors (57), residual `<asp>`/triple-tick fragments, and empty-program cases (39) indicate that even with `extract_code_blocks`, fences and tags occasionally bleed into the program text.
4. **`#mod` still appears.** Even though the autoresearch branch added a `#mod → \` regex, 36 lexer errors for `#mod` show up in audits — meaning either the fix isn't applied on `feature/baseline`, or it's applied late (after a different Clingo error masks it). Baseline has *no* programmatic syntax repair.
5. **No multi-candidate diversity on baseline.** A single sample at temperature 1.0 means one syntactic mistake terminates the whole puzzle. Agentic-nemotron showed that 6 diverse candidates plus per-candidate patching dramatically increases the odds that *one* program is at least syntactically clean.
6. **Refinement attempts are wasted on syntax loops.** `MAX_ATTEMPTS=6` is consumed by syntax-only churn before the model ever sees a semantic-feedback round.
7. **Thinking budget is unmanaged.** `MAX_TOKENS=200_000` with `THINKING=True` and reasoning_budget hints ignored by Nemotron — the model spends most of its budget thinking and sometimes truncates the response (empty programs, 39 cases).

### What `feature/agentic-nemotron` and `autoresearch/apr24` already taught us

Worth reusing on baseline (in priority order):
- `quick_syntax_fix` deterministic regex patches (7 rules: `#mod→\`, aggregate-on-left swap, `#count(...)`→`#count{...}`, SWI-Prolog `aggregate/3` and `aggregate_all/3`, `#choose→#min`, `exactly N {}` form, uppercase `#const`).
- Robust code extraction (handles `<asp>`, `<clingo>`, ``` ```asp ```, ``` ```clingo ```, plain triple-tick, unclosed variants, full-response fallback).
- 6 diverse candidate strategies (default, choice+constraints, deterministic, row-by-row, column-by-column, pattern-explicit) selected by `solved > clean > accuracy`.
- Single-shot LLM `rewrite_syntax_fix` with the syntax guide injected.

Worth NOT reusing:
- `reasoning_budget=4096` (autoresearch: 0/20, ignored by model).
- 5-stage decomposed prompts (worse than 2-prompt initial+reattempt).
- Full `content`-rewrite tool in `edit_code` (empirically destroyed logic).

## Ranked single-PR experiments (build on `feature/baseline`)

Each experiment is independent, scoped to one PR, has a clear hypothesis, a single change surface, and a measurable outcome. Run order is by expected ROI per engineering hour.

### E1 — Deterministic syntax repair pre-Clingo (port `quick_syntax_fix`)
- **Hypothesis:** Mechanical regex fixes can convert ~30–40% of clingo_error verifications into either clean programs or different (more semantic) errors, without touching the LLM.
- **Change:** Add `src/utils/syntax_repair.py` with the 7 rules from `feature/agentic-nemotron:src/agent.py:quick_syntax_fix()`. Apply in `agent.generate_initial`/`generate_reattempt` after `extract_code_blocks` and before Clingo verification. Log before/after diff per attempt.
- **Files:** new `src/utils/syntax_repair.py`; `src/agent/agent.py` (call site); `src/utils/grids.py` (extraction hardening — strip backticks, balance fences).
- **Metric:** Drop in unsafe-variable + unexpected-`=` + `#mod` lexer + backtick-lexer counts in audits. Same 20 puzzles as baseline runs.
- **Cost:** ~2–3 hours.

### E2 — Multi-candidate generation with simple ranking
- **Hypothesis:** Generating N=4 candidates per puzzle and keeping the best by `(solved, syntax_clean, train_accuracy)` will ~double the initial-pass success rate, simply by sampling around the syntactically-bad regions.
- **Change:** Add `N_CANDIDATES=4` to `config.py`. In `agent.generate_initial`, call `engine.generate_batch` with N copies of each prompt (flat batch, single vLLM pass). Verify each on training examples; pick best with the ranking from agentic-nemotron's `_score_candidates`. Store all candidates in the audit JSON.
- **Files:** `src/config/config.py`, `src/agent/agent.py`, `src/main.py` (initial step), `src/utils/session.py` (record candidates), `streamlit-interface/app.py` (show them).
- **Metric:** Fraction of puzzles where the best candidate is syntax-clean after E1. Wall-clock cost per puzzle.
- **Cost:** ~3–4 hours. Depends on E1 to be useful.

### E3 — Concept-level error feedback in the reattempt prompt
- **Hypothesis:** Replacing raw Clingo errors with *categorized, concept-level* feedback ("aggregate-on-left: move `#min{...} = X` to `X = #min{...}`") will let refinement actually escape syntax loops (today: 82% stuck).
- **Change:** Build `src/utils/error_classifier.py` mapping the buckets in the diagnosis table to short prose explanations + a corrected fragment when E1's regex would fix it (so the model sees the rule). Inject into the `==HISTORY==` slot of `prompts/reattempt.txt`. Keep raw Clingo error appended as fallback.
- **Files:** new `src/utils/error_classifier.py`; `src/agent/utils.py` (`build_reattempt_user`); `src/prompts/reattempt.txt`.
- **Metric:** Fraction of `clingo_error → not-clingo_error` transitions across refinement attempts. Today: 47/361 (13%). Target: ≥35%.
- **Cost:** ~3 hours.

### E4 — Inject the curated Clingo syntax guide on rewrite only
- **Hypothesis:** Putting the full 33KB `docs/docs.clingo_syntax_guide.md` in the system prompt every turn pollutes context and hurts. Putting it only in a dedicated single-shot rewrite prompt — triggered when a candidate has a syntax error — gives the model the relevant material exactly when it needs it.
- **Change:** Add `prompts/rewrite.txt` (system message: "rewrite the program so it's Clingo-valid, preserve the logic"; embed full guide). Add `agent.rewrite_syntax_fix(program, errors)` ported from `feature/agentic-nemotron`. Run after E1 if syntax errors remain. Cap at 2 LLM rounds.
- **Files:** new `src/prompts/rewrite.txt`, `src/agent/agent.py`, `src/main.py`.
- **Metric:** Of candidates that still have syntax errors after E1, fraction made clean by rewrite. Token budget per puzzle.
- **Cost:** ~3 hours. Depends on E1.

### E5 — Tighten code extraction and reject markup-leak programs
- **Hypothesis:** A small but stubborn slice of failures (backtick-lexer 57, empty 39, `unexpected <` 107) come from `<asp>`/fence parsing edge cases. Hardening extraction kills these without touching the LLM.
- **Change:** Rewrite `extract_code_blocks` to: prefer last `<asp>...</asp>`, fall back to last fenced block (any of `asp|clingo|prolog|""`), strip trailing/leading backticks, drop XML-tag-only lines, validate non-empty; if empty, return `""` and log a clear "extraction failed" diagnostic instead of garbage.
- **Files:** `src/utils/grids.py`.
- **Metric:** Backtick lexer count and empty-program count in audits should both go to ~0.
- **Cost:** ~2 hours.

### E6 — Raise refinement attempts only after E1+E3
- **Hypothesis:** `MAX_ATTEMPTS=6` is correctly sized *if* refinement actually progresses; raising it without E1/E3 just buys more identical syntax-error rounds.
- **Change:** After E1+E3 land, bump `MAX_ATTEMPTS` to 10 *and* add an early-stop: if the last 2 attempts produced the same normalized error signature, break the loop. Optionally bump candidate count or temperature on the next attempt to break the loop.
- **Files:** `src/config/config.py`, `src/main.py` (early-stop logic), `src/utils/error_classifier.py` (signature).
- **Metric:** Solve rate vs. attempts. Token cost per solved puzzle.
- **Cost:** ~2 hours. Sequenced after E1+E3.

### E7 — Constrain decoding for the program block
- **Hypothesis:** A surprising fraction of syntax errors are LLM artifacts (markdown, prose mid-program, repeated backticks). Constrained decoding via vLLM's `guided_decoding` with a Lark grammar covering a useful subset of Clingo (facts, normal rules, choice rules, integrity constraints, comparisons, the four common aggregates) would eliminate categories 4, 5, 7, 8 wholesale at the cost of expressiveness.
- **Change:** Write a permissive Lark grammar for "Clingo-without-meta-statements". Wrap the final ASP block in `<asp>...</asp>` and apply `GuidedDecodingParams(grammar=...)` only to the span between those tags using vLLM logits processors. Keep an unconstrained fallback for the thinking section.
- **Files:** new `src/agent/grammar.py`, `src/agent/vllm_engine.py`.
- **Metric:** Lexer/syntax error count drops to ~0 *for the constrained span*. Watch for over-constraint regressions in semantic accuracy.
- **Cost:** ~1–2 days. Higher risk; do after E1–E5 to know the grammar is the bottleneck.

### E8 — Skeleton-then-body, deterministic preamble
- **Hypothesis:** A large share of unsafe-variable errors come from the model writing custom dimension/coord helpers wrong. Always prepending a known-good preamble (`output_cell/2`, dimensions via `#count`, coordinate ranges, color choice rule) and asking the LLM only for the *transformation rules* shrinks the surface area where it can break.
- **Change:** Add a fixed preamble template in `prompts/`. Modify the initial prompt: "the preamble below is fixed; you only need to write the transformation rules under `% --- TRANSFORMATION ---`." Concatenate preamble + LLM body before Clingo runs.
- **Files:** new `src/prompts/preamble.lp`, `src/agent/agent.py`, `src/prompts/initial_prompt.txt`.
- **Metric:** Unsafe-variable count, plus solve rate. Risk: preamble may be wrong for some puzzle types; keep one strategy in E2 that opts out of the preamble.
- **Cost:** ~3 hours. Best evaluated *after* E2 so we can run it as one of the candidates.

### Lower-priority probes (cheap, run as background)
- **P1:** Drop `THINKING=True` for one cohort (autoresearch tried, reverted in 1 commit — re-test now that other things have changed; metric: same solve rate at 1/10th tokens?).
- **P2:** Lower `TEMPERATURE` from 1.0 → 0.7 for one cohort; multi-candidate (E2) probably wants higher temperature, single-candidate probably wants lower.
- **P3:** Add `#show output/3.` to the preamble unconditionally — small change, eliminates a class of "answer set printed but not the right predicate" failures (autoresearch did this in `14048e1`).
- **P4:** Per-error `<wrong>/<right>` few-shot block at the bottom of the system prompt for the top 6 buckets, *only* in the rewrite prompt (E4) — keep initial prompt lean.

## Verification plan

For each experiment, run a fixed evaluation cohort and compare against a baseline run with identical seed and puzzle set:

1. **Cohort:** the same 20-puzzle ARC-v1-training subset used in `src/audit/20260423_184218/` and `src/audit/20260424_222615/` (already audited; gives apples-to-apples comparison).
2. **Per-experiment run:**
   - `sbatch run.job` with the experiment's branch/config; SLURM job ID logged in `docs/`.
   - Save audit dir under `src/audit/<run_id>/`.
   - Inspect via the Streamlit inspector (`streamlit-interface/app.py`).
3. **Metrics to record per run** (small `scripts/audit_stats.py` to compute, ~30 lines):
   - `final_correct` puzzles / total.
   - Per-attempt verification status histogram (`clingo_error / unsatisfiable / wrong_values / correct`).
   - Refinement transition matrix (today: 82% `clingo_error → clingo_error`).
   - Top-10 normalized Clingo error patterns + counts.
   - Tokens used per puzzle (initial + refinement).
4. **Decision rule:** an experiment "wins" if it improves at least one of {solve rate, syntax-clean rate, `clingo_error → not-clingo_error` transition rate} by ≥10% relative without a regression in the others.
5. **Documentation:** write a one-page `docs/experiments/<eXX>.md` per experiment with the hypothesis, change, metrics, raw numbers, and verdict (kept / reverted). This doubles as the autoresearch progress log.
