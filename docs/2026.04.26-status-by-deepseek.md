# Synthesis: ASP-ARC Pipeline — Error Patterns, Bottlenecks, and Experiments

## 1. What Problems the Agent Encounters

The core failure mode is a **syntax-truncation death spiral**: the LLM generates syntactically invalid Clingo programs, which prevents any logical evaluation, which means the refinement loop never gets meaningful feedback, which means the next attempt also produces invalid syntax.

From the audit data across baseline runs (Qwen3.6-35B) and agentic-nemotron runs:
- **~40-50% of initial programs** have at least one Clingo parse/ground error
- **Puzzle `39e1d7f9`**: All 6 refinement attempts failed with syntax errors — never escaped to logic evaluation
- **Puzzle `8a004b2b`**: All 6 refinement attempts failed with syntax errors — same pattern
- **Puzzle `8d510a79`**: 5 of 6 refinement attempts were syntax errors; only one hit "0 answer sets" (logic error)
- Even the solve on attempt 5 in run 21445746 (agentic branch) came *after* 4 attempts that each hit different syntax/ground errors

The secondary failure mode is **logical stagnation**: when the program parses but produces wrong outputs, the model gets stuck making semantically equivalent rewrites without fixing the core logic. Puzzle `0520fde7` exemplifies this — 16+ attempts across two runs, always `wrong_values`, never improved.

## 2. Most Common Syntax Mistakes (Ranked by Frequency in Audit Data)

### Tier 1 — High Frequency (seen in nearly every run)

| # | Error | Example | Clingo message |
|---|-------|---------|----------------|
| 1 | **Aggregate on left of `=`** | `#min{R:p(R)} = X` | `syntax error, unexpected =, expecting ) or ;` |
| 2 | **Unsafe variables under negation** | `missing(R) :- col(C), not input(R,C,_).` (R ungrounded) | `unsafe variables in: ...` |
| 3 | **Predicate names in arithmetic** | `R = row_min - 1` (row_min is a predicate, not value) | `syntax error, unexpected =` or grounding error |
| 4 | **Top-level assignment without head** | `MinR = #min{R : p(R)}.` | `syntax error` or grounding error |
| 5 | **`#mod` instead of `\`** | `C #mod 2 = 0` | `lexer error, unexpected #mod` |

### Tier 2 — Medium Frequency

| # | Error | Example | Clingo message |
|---|-------|---------|----------------|
| 6 | **`#count(...)` with parens not braces** | `#count(X : pred(X))` instead of `#count{ X : pred(X) }` | `unexpected (, expecting {` |
| 7 | **`aggregate/3` (SWI-Prolog)** | `aggregate(N=#count, input(R,C,V), N)` | `syntax error, unexpected =` |
| 8 | **`{ } exactly N` suffix** | `{ output(R,C,V) : color(V) } exactly 1 :- cell(R,C).` | `unexpected <IDENTIFIER>, expecting )` |
| 9 | **Uppercase `#const` name** | `#const MAX = 30.` | `unexpected <VARIABLE>, expecting <IDENTIFIER>` |
| 10 | **Bare comma-separated atoms without head** | `input(R,C,V), V != 0.` (no `:-`) | Various parse errors |

### Tier 3 — Lower Frequency (seen in audit data)

| # | Error | Example |
|---|-------|---------|
| 11 | **Inline aggregate-to-aggregate comparison** | `#count{R : input(R,P,5)} = #count{R : row(R)}` in body |
| 12 | **Missing `#show` or extra predicate atoms polluting answer set** | Causes `underconstrained` or wrong grid reconstruction |
| 13 | **Mangled syntax from code extraction failure** | Thinking text or half-extracted code passed as program |
| 14 | **`#choose` (non-existent)** | Model invents `#choose{...}` — likely means `#min` |
| 15 | **`#sum{} = Var` in constraint body** | Same as aggregate-on-left pattern but with `#sum` |

## 3. What's Holding It Back — The Root Causes

### A. The model doesn't know Clingo syntax with precision

This is the dominant constraint. The Qwen3.6 model (baseline branch) and even Nemotron-Cascade-2 (agentic branch) both consistently produce SWI-Prolog-isms, inverted aggregates, and unsafe variable patterns. The few-shot examples in the prompt help but don't cover all error modes. The syntax guide helps but the model doesn't internalize it in one read — it needs to see the fix *applied to its own code*.

### B. Syntax errors are treated as feedback, not as fixable

In the baseline, when Clingo returns a parse error, the refinement loop feeds that raw error back as text and asks the model to "fix it." But the model's fix attempt is a full program rewrite, not a surgical patch — and the rewrite often introduces *new* syntax errors while fixing the old ones. The agentic branch's key insight: separate syntax fixing from logic refinement.

### C. Code extraction is fragile

Both branches have sophisticated `extract_code_blocks()` functions with multiple fallback strategies, but:
- Model sometimes wraps code in `<asp>` tags, sometimes in ```` ``` ```` fences, sometimes both
- Thinking mode means the response can contain code examples inside the thinking block that are NOT the final answer
- Empty programs (0 chars) appear frequently — the model's thinking consumed the token budget and the response was empty/truncated
- The `thinking_code_extraction` issue doc confirms extraction failures when both `<asp>` and ```` ``` `````` patterns appear

### D. The thinking budget is not controlled (baseline branch)

The baseline has `MAX_TOKENS=200000` and `THINKING=True` but no `reasoning_budget`. The model sometimes thinks for 80K+ tokens then produces empty output. The autoresearch branch added `REASONING_BUDGET=4096` to cap thinking, which the agentic branch also adopts implicitly by the prompt design.

### E. Multi-candidate generation is not exploited (baseline)

The baseline generates exactly 1 program per puzzle. The autoresearch branch added `N_CANDIDATES=3` and the agentic branch uses `N_CANDIDATES=6` with diverse prompt strategies. The empirical evidence in the agentic branch docs shows this dramatically improves solve rates — simply having more shots at correct syntax.

### F. No syntax-fix pipeline in refinement (baseline)

The baseline applies zero syntax repair to refinement outputs. The agentic branch applies `quick_syntax_fix` + `rewrite_syntax_fix` after each refinement output. This prevents one trivial syntax error from wasting an entire refinement round.

### G. Async refinement and batching improvements help throughput but not solve rate

The agentic branch's async refinement and flat-batch candidate generation are engineering optimizations that improve wall-clock time and GPU utilization. They don't change whether the model produces correct ASP — they just let you run more attempts in the same time.

## 4. What Experiments Could Improve on This

### High-Priority Experiments

**1. Drop-in replacement: swap the existing prompt's few-shot examples with the Clingo Syntax Guide as the primary in-context reference.**

Right now, the initial prompt has two complex worked examples (tiling and flood-fill) that take ~250 lines. These are cognitively expensive and error-prone examples. The autoresearch branch tried replacing them with a simple "copy input to output" example and a fill-in-the-blanks template (`b685651`). The comprehensive syntax guide you wrote (`docs/docs.clingo_syntax_guide.md`) is 251 lines of pure Clingo reference material. **Experiment**: make the system prompt be the syntax guide (or a condensed 8-error "cheat sheet" version) plus exactly one simple worked example, and see if syntactic validity improves. The agentic branch proved that having the syntax guide *available during rewrite* fixes errors — the question is whether having it *as the primary reference during initial generation* prevents errors from occurring in the first place.

**2. Two-pass generation: first pass = logic description in natural language, second pass = ASP encoding from that description.**

The agentic-nemotron `single_step.txt` prompt asks the model to "in your reasoning, identify the transformation rule... then write the complete ASP program." But the model's thinking is discarded before code extraction. **Experiment**: add a structured intermediate step: the model outputs `<analysis>` describing the rule in natural language, then the system feeds that analysis back with a second call: "Now translate this analysis to ASP." This gives the model a chance to correct logical errors before committing to syntax. The `architecture.md` from an earlier version of the agentic branch described exactly this STEP 1-4 pipeline that was later abandoned in favor of single-shot.

**3. Syntactic pre-validation via a second, cheaper LLM call that ONLY checks syntax.**

The `rewrite_syntax_fix` in the agentic branch works: a focused single-shot LLM call that fixes only syntax (not logic) resolves most parse errors in 1-3 rounds. **Experiment**: run this *before* the initial candidate evaluation, for every candidate, every time. The agentic branch already does this, but only after selection — do it earlier to increase the pool of evaluable candidates. Combine with the autoresearch branch's `repair_asp_syntax()` regex fixes as a pre-pass.

**4. Deterministic syntax repair: expand `repair_asp_syntax()` / `quick_syntax_fix()` to catch more error classes without LLM calls.**

The autoresearch branch's `repair_asp_syntax()` handles: `#mod→\`, comma-separated atoms → integrity constraints, stray `:-` removal, aggregate-on-left fix, and `#choose→#min`. The agentic branch's `quick_syntax_fix()` adds: `exactly N → N{}N`, `#const UPPER → lower`, aggregate parens→braces, and SWI-Prolog `aggregate/3` and `aggregate_all/3` fixes. **Experiment**: merge the best of both into a comprehensive regex repair that handles the top 8 error classes, and measure how many programs it fully fixes without any LLM involvement. The audit data shows that `syntax error, unexpected =, expecting ) or ;` (aggregate-on-left) is the single most common error — fixing this one deterministically would clear ~30-40% of failures.

**5. Run both models (Qwen3.6 and Nemotron-Cascade-2) on the same puzzle set and compare syntax error rates.**

The baseline branch uses Qwen3.6-35B. The agentic branch uses Nemotron-Cascade-2-30B. The error patterns appear similar but the models have different vocabularies and training data. **Experiment**: a controlled A/B test — same prompts, same puzzles, different models — to see which has the lower syntax error rate out of the box. This informs whether a model swap (or ensemble) would help more than pipeline improvements.

### Medium-Priority Experiments

**6. Add `#show output/3.` enforcement.**

The comprehensive syntax guide mentions `#show` but neither branch's prompts require it. Many programs produce answer sets with hundreds of auxiliary atoms that confuse grid reconstruction. **Experiment**: add `#show output/3.` to the prompt template and/or automatically append it to every program before Clingo evaluation.

**7. Stricter answer-set validation: require exactly 1 answer set.**

The current verification treats 0 answer sets as "unsatisfiable" error and >1 as "underconstrained" with useful feedback. But the model doesn't get penalized during selection for underconstrained programs — they get a diff based on the first answer set and may score well by accident. **Experiment**: add a heavy accuracy penalty for programs that don't produce exactly 1 answer set during candidate ranking. This forces the model to write complete specifications, not underconstrained guesses.

**8. Syntax-then-logic refinement: split the reattempt prompt into two stages.**

The agentic branch's reattempt prompt mixes syntax advice with logic advice: "CLINGO ERROR — Fix the specific syntax error" alongside "WRONG VALUES — Logic is incorrect." The model may fix syntax but break logic, or vice versa. **Experiment**: if a refinement attempt has a syntax error, run a dedicated syntax-fix pass first (like agentic branch does), then feed the syntax-clean program into the logic refinement prompt. Ensure the logic phase never sees syntax errors.

**9. Self-consistency voting across multiple candidates on test predictions.**

The agentic branch generates 6 candidates but only selects 1. The `arc-agi-harness.md` doc describes a voting mechanism: if 3 of 6 candidates produce the same test output, that agreement is strong evidence of correctness. **Experiment**: run all 6 syntax-cleaned candidates on the test inputs and check for agreement, rather than selecting just one.

### Lower-Priority Experiments

**10. Temperature sweep: find the sweet spot for ASP generation.**

The baseline uses `TEMPERATURE=1.0`, `TOP_P=0.95`. The agentic branch uses the same. For structured code generation, lower temperature often produces more syntactically valid output but less logical diversity. **Experiment**: run the same puzzle set at temperatures 0.3, 0.6, 1.0 to compare syntax error rate vs. logical correctness.

**11. Post-generation Clingo compiler as a final guard.**

Add a step after the entire pipeline that runs the final program through Clingo, captures the exact error, feeds it to the syntax rewrite LLM (with the syntax guide in context), and repeats until clean or 3 rounds. This is already effectively what the agentic branch does before refinement — but do it as a final post-pass on the program that gets saved.

**12. Line-level error annotation.**

The agentic branch's `_annotate_clingo_error()` adds hints to cryptic Clingo errors. The comprehensive syntax guide is much more detailed. **Experiment**: use the syntax guide as a lookup table: for each Clingo error pattern, return the specific correction from the guide. For example, `unexpected =` on a line containing `#min` or `#count` → inject "Hint: aggregates must be on the right-hand side of `=`."
