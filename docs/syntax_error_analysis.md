# Syntax Error Analysis

Analysis of syntax errors in ASP programs generated by Nemotron-Cascade-2 across
5 seeds (11, 23, 47, 91, 137), each running 20 ARC-AGI puzzles with 6 candidate
generation strategies.

## Methodology

### Data Source

Result files live under `results/seed<N>_<run_id>/` as per-puzzle JSONs. Each
contains a `candidates[]` array with `syntax_fix_stages` and `syntax_fix_details`
tracking every fix attempt, plus a `syntax_agent` object tracking the puzzle-level
post-hoc rewrite.

### How the Pipeline Works

```
generate(N candidates) ──► quick_syntax_fix ──► syntax check
                            │                        │
                            │                   ┌────┘
                            ▼                    ▼
                       passes?  ◄─── batch LLM rewrite (≤3 rounds)
                            │
                     ┌──────┴──────┐
                     ▼              ▼
                 evaluate      evaluate
                 (clean)       (may have syntax errors)
                     │              │
                     └──────┬──────┘
                            ▼
                   pick best candidate
                            │
                     ┌──────┴──────┐
                     ▼              ▼
                syntax OK? ──► puzzle-level syntax agent
                     │              │
                     ▼              ▼
                evaluate test  async refinement loop
```

### Counting Rules

- **Initial syntax clean**: candidate had `syntax_fix_stages = []` or
  `['quick_fix']` — no LLM rewrite was needed.
- **Fixed by LLM rewrite**: candidate had `'rewrite'` in `syntax_fix_stages` and
  no `'rewrite_partial'`.
- **Still broken after LLM**: candidate had `'rewrite_partial'` in
  `syntax_fix_stages`.

Error classification uses regex matching on Clingo error messages from the
`rewrite_details[].syntax_error_before` and `syntax_error_after` fields.

### Running the Analysis

```
# Aggregate across runs:
python scripts/summarize_arc_runs.py results/seed11_* results/seed23_* ...

# Full error taxonomy:
python scripts/analyze_syntax.py results/seed11_* results/seed23_* ...
```

## Results

### 1. Generation Syntax Quality

Across 509 evaluated candidates (out of 600 max — puzzles that solve early
skip remaining candidates):

| Metric | Count | Percent |
|--------|-------|---------|
| Syntax-clean from model | 26 | 5.1% |
| Has syntax errors | 483 | 94.9% |
| Of which: fixed by LLM rewrite | 175 | 36.2% of broken |
| Of which: still broken after 3 LLM rounds | 308 | 63.8% of broken |

The model generates syntactically invalid ASP roughly **95% of the time**.

### 2. Error Type Taxonomy (Before LLM Rewrite)

Errors extracted from the first syntax check of every broken candidate:

| Error Type | Count | Share |
|------------|-------|-------|
| `unexpected_hash` | 579 | 41.8% |
| `unsafe_variable` | 445 | 32.1% |
| `unexpected_other` | 203 | 14.6% |
| `unexpected_colon` | 126 | 9.1% |
| `unexpected_bang` | 24 | 1.7% |
| Other | 9 | 0.6% |
| **Total** | **1386** | |

The `#` character used in Clingo aggregates (`#count`, `#sum`, `#min`, `#max`,
`#true`, `#false`) is the most common source of parse failures.

### 3. Error Types After LLM Rewrite (Persistent Errors)

Errors in the last failed rewrite round:

| Error Type | Count | Share |
|------------|-------|-------|
| `unsafe_variable` | 558 | 46.1% |
| `unexpected_hash` | 394 | 32.5% |
| `unexpected_other` | 168 | 13.9% |
| `unexpected_colon` | 68 | 5.6% |
| `unexpected_bang` | 18 | 1.5% |
| Other | 5 | 0.4% |
| **Total** | **1211** | |

Key observation: `unsafe_variable` **increases** from 32.1% → 46.1% as the
LLM replaces `#`-related constructs with hand-written logic that introduces
free variables.

### 4. Fix Round Distribution

| Rounds Used | Candidates | Share |
|-------------|------------|-------|
| 1 | 37 | 7.3% |
| 2 | 67 | 13.2% |
| 3 | 405 | 79.6% |

Most candidates need all 3 rounds. Few are fixed early.

### 5. Strategy Comparison

| Strategy | Total | Clean | Broken | Fixed | Stays | Clean% | Fix% |
|----------|-------|-------|--------|-------|-------|--------|------|
| 0 | 100 | 4 | 98 | 40 | 58 | 4.0% | 40.8% |
| 1 | 92 | 8 | 88 | 30 | 58 | 8.7% | 34.1% |
| 2 | 90 | 18 | 81 | 29 | 52 | 20.0% | 35.8% |
| 3 | 85 | 2 | 84 | 24 | 60 | 2.4% | 28.6% |
| 4 | 84 | 12 | 78 | 21 | 57 | 14.3% | 26.9% |
| 5 | 84 | 8 | 80 | 31 | 49 | 9.5% | 38.8% |

Strategy 2 generates the cleanest syntax (20%). Strategy 0 is best at
fixing broken programs (40.8%). Strategy 4 is the worst at fixing (26.9%)
and strategy 3 generates the dirtiest syntax (2.4% clean).

### 6. Accuracy of LLM-Fixed Candidates

Of the 175 candidates whose syntax was fixed by the LLM rewrite:
- **138 (79%)** had at least 1 correct training example
- The remaining 21% produced valid Clingo but completely wrong semantics

### 7. Puzzle-Level Syntax Agent

Only **8% of puzzles** selected a best candidate that still had syntax errors,
triggering the per-puzzle syntax agent:

- `rewrite_clean`: 3 — agent succeeded in fixing on final attempt
- `stalled`: 2 — agent hit tool call errors
- `syntax_clean`: 2 — agent's first output was clean
- `max_attempts`: 1 — agent used all 3 rounds without fixing
